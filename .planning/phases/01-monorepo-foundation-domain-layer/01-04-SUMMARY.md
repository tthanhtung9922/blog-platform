## Decisions Made

- **MediatR allowed in two Domain namespaces, not one:** The original plan said "only Blog.Domain.Common may reference MediatR." In practice, domain event records in `Blog.Domain.DomainEvents` also reference MediatR because they implement `IDomainEvent : MediatR.INotification`. NetArchTest uses IL-level dependency detection, not import-statement analysis — so any class that implements an interface which inherits a MediatR type counts as "having MediatR dependency." The test was adjusted to reflect this accurate architectural reality.

- **Renamed the MediatR boundary test:** Changed from `Domain_OnlyAllowedExternalDependency_IsMediatR` to `Domain_AggregatiesAndValueObjects_ShouldNot_ReferenceMediatRDirectly`. The new name accurately describes what the test verifies: that aggregates, value objects, repositories, and exceptions do not directly implement or inherit MediatR types.

- **AggregateRoot inheritance uses explicit type list:** Enumerating `{Post, Comment, User, Tag}` rather than scanning all classes in `Blog.Domain.Aggregates.*` prevents false failures from child entities (`PostContent`, `PostVersion`) that live in the same namespace but are not aggregate roots.

**1. [Rule 1 - Bug] MediatR boundary test logic corrected for domain event namespace**
- **Found during:** Task 1 (running tests after initial implementation)
- **Issue:** The plan's original test `Domain_OnlyAllowedExternalDependency_IsMediatR` checked that only types in `Blog.Domain.Common` have MediatR dependency, using `.ResideInNamespace("Blog.Domain.Common")`.
